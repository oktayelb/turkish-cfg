# ML Model: Contrastive Morphological Ranking

## Overview

The ML layer (`ml/ml_ranking_model.py`) ranks complete morphological analysis candidates. The rule-based decomposer still generates possible root + suffix chains; the model now learns directly from the choice the user or treebank says is correct.

The training objective is no longer masked suffix prediction. Each training item is a candidate set:

```
[gold sentence analysis, wrong analysis 1, wrong analysis 2, ...]
```

The model assigns one scalar score to each complete candidate and is trained with cross-entropy so the gold candidate has the highest score.

## Architecture

`SentenceDisambiguator` is a Transformer encoder over flattened suffix-token sequences.

Each token position combines:

- suffix or closed-class token id
- noun/verb/special/closed-class category
- suffix group id
- `comes_to` type id
- `makes` type id
- position within the word
- word-final marker
- absolute sequence position

The concatenated embeddings are projected to the model dimension, passed through the Transformer, mean-pooled over non-padding tokens, and fed to `rank_head`:

```
candidate sequence -> Transformer -> mean pool -> rank_head -> scalar score
```

The older `lm_head` remains in the module for checkpoint compatibility, but ranking training and inference use `rank_head`.

## Sequence Encoding

`build_sentence_sequence()` flattens a list of per-word suffix chains:

```
evlerden geldim
ev:  [plural_ler, ablative_den, WORD_SEP]
gel: [pasttense_di, conjugation_1sg, WORD_SEP]
```

Bare-root words contribute only a word boundary and receive a small configurable prior penalty during scoring.

## Training

### Interactive Word Training

For a selected word candidate:

1. The selected analysis is encoded as the gold candidate.
2. Other decomposer candidates for the same word become negatives.
3. `Trainer.train_sentence()` ranks the gold candidate against those negatives.

### Interactive Sentence Training

For a confirmed sentence:

1. The selected candidate for every word forms the gold sentence analysis.
2. Negatives are generated by single-word substitution: keep the gold sentence fixed, replace one word with a wrong decomposer candidate.
3. The number of negatives is capped by `config.max_negative_candidates`.
4. The model is trained to score the gold sentence above every negative.

This creates many useful training comparisons from one confirmed sentence without exploding into the full Cartesian product.

### Bulk Relearning

`relearn` reads logged/treebank entries, then for each entry:

1. Runs the decomposer again for each logged word.
2. Matches the logged suffix chain to one decomposer candidate.
3. Uses that match as gold.
4. Generates single-substitution negatives from the remaining decomposer candidates.
5. Trains on candidate sets in shuffled mini-batches.

If a logged chain cannot be matched because the decomposer changed, the code falls back to direct suffix-name encoding for the gold chain and skips negatives for that item.

### Curriculum Training

`curriculum` is dynamic hard-negative training. It is designed to continue from an existing checkpoint, not to replace ordinary warm-up training from scratch.

The static `relearn` path creates candidate sets from decomposer alternatives and trains against those sets repeatedly. This is useful, but after several epochs the model may learn the fixed negative set well enough that additional epochs provide less new signal. Curriculum training changes the negative set between training rounds by using the current model as the miner.

The loop is:

1. Load the current `ml/model.pt` checkpoint.
2. Optionally run a static warm-up phase using the same candidate sets as `relearn`.
3. For each curriculum generation:
   - reload logged/user/treebank gold entries
   - run the decomposer for each logged word
   - match the logged gold chain to the generated candidates
   - build a wide pool of single-substitution wrong analyses
   - score those wrong analyses with the current model using `score_flat_sequences()`
   - select the wrong analyses the current model finds most tempting
   - train for a short number of epochs on the newly mined candidate sets
   - save the updated checkpoint

Each mined candidate set still has the same shape:

```
[gold sentence analysis, mined wrong analysis 1, mined wrong analysis 2, ...]
```

Candidate `0` is always the gold analysis. The mined negatives are wrong analyses that the current model scored highly or that were sampled as medium/easy controls.

#### Hard, Medium, and Easy Negatives

The miner does not keep only the absolute hardest negatives. Keeping only the hardest cases can over-focus training on confusing edge cases and make the model forget easier distinctions.

The default mix is configured in `ml/config.py`:

- `hard_negative_count = 6`: highest-scoring wrong analyses
- `medium_negative_count = 2`: randomly selected middle-ranked wrong analyses
- `easy_negative_count = 2`: low-scoring wrong analyses
- `max_negative_candidates = 10`: total negatives per candidate set
- `dynamic_negative_pool_size = 100`: maximum wrong single-substitution analyses scored before selection

This means a typical curriculum candidate set contains one gold analysis and up to ten wrong analyses. The wrong analyses are mostly hard negatives, with a small safety mix of medium and easy negatives.

#### Why Single-Substitution Negatives

For a sentence, the full Cartesian product of every word candidate can become very large. SAV-YAR avoids that explosion by keeping the gold sentence fixed and replacing one word at a time with a wrong candidate. This produces plausible, targeted sentence-level negatives while keeping GPU memory bounded.

#### How to Run

```text
curriculum
```

The output is an updated checkpoint, not a printed hard-example report. Use validation `RankAcc`, `margin`, and `rank_loss` to decide whether the curriculum is improving the model.

## Loss and Metrics

For each candidate set, candidate `0` is gold:

```
scores = model(candidate_set)
loss = CrossEntropy(scores, target=0)
```

Bulk training reports:

- `loss`: cross-entropy over candidate sets
- `RankAcc`: fraction of candidate sets where gold has the highest score
- `margin`: average `score(gold) - best_score(negative)`

Validation uses the same candidate-ranking objective.

## Inference

Inference also uses scalar candidate scores:

- `score_sentence_chains()` scores a complete sentence analysis.
- `score_candidates()` scores alternatives for one word in context.
- `get_top_sentence_predictions()` performs beam search over sentence candidates and keeps the highest-scoring beams.

Higher score means the candidate is preferred.

## Checkpointing

Checkpoints store model weights, optimizer state, scheduler state, histories, global step, replay buffer, and suffix inventory. Older checkpoints without `rank_head` can still load because incompatible or missing parameters are ignored and newly added parameters keep their initialized values.

## What the Model Learns

The model now learns disambiguation preferences directly:

- prefer the gold suffix chain over decomposer alternatives
- reject common confusions such as `locative_de` vs `ablative_den`
- penalize plausible but wrong possessive/case boundaries
- use surrounding sentence morphology when ranking a candidate

It still does not see roots or surface characters directly; lexical information enters only through the decomposer candidate set and the logged gold choice.
